3.2 Snapshot wiring

_member_settings(profile) sources values from the member's YAML/Markdown frontmatter dict (<lab-mgmt>/members/<handle>.md). It looks for an optional obsidian: sub-block, plus the existing contact: and location: blocks; falls back to legacy flat keys (obsidian_vault_path, etc.) when present. Notebook and oracle subfolder names default to lab-notebook and oracle when unset.

3.3 UI: MemberProfileModal

A single scrollable overlay with four sections, in order:

SectionModeFields
Identity read-only handle, name, role, lab, vault (short form)
Obsidian & notebook editable vault path (full), vault name, notebook subfolder, oracle subfolder
Contact editable email, ORCID, Bluesky, GitHub, OSF, website
Location editable office, dry lab, wet labs, address, city, department

Local form state is initialised from D.member_settings via useState. Save POSTs JSON to /api/member/settings; on 4xx/5xx the modal stays open and surfaces the status code as a muted hint, on network failure shows "offline — local only". Close dismisses without saving; clicks on the dimmed backdrop also dismiss (matches every other modal in the dashboard).

3.4 FooterMeta conversion

Converted the plain FooterMeta() function into a stateful one. Added const [showProfile, setShowProfile] = useState(false); near the top, then inside the <div className="footer-meta">:

  • Modal mount: {showProfile && <MemberProfileModal onClose={() => setShowProfile(false)} />}
  • Member name + a small ⚙ gear button rendered as a sibling, both wrapped in an inline-flex span; clicking the gear opens the modal.
  • The handle/role line is preserved as a muted mono row underneath.
  • The original Location <h5> is reattached after the new header block; all subsequent contact / affiliations sub-grids are unchanged.

6 · Hand-off notes

  • The POST /api/member/settings endpoint is referenced by the modal but not yet implemented on the backend. The modal handles 4xx/5xx and network errors gracefully — adding the FastAPI route that writes <lab-mgmt>/members/<handle>.md frontmatter is the obvious next phase.
  • _personal_oracle reads from ~/.claude/agent-memory/oracle/. When a user has switched their personal Oracle to live in their Obsidian vault directly, this path resolution should follow the notebook_actions.notebook_folder() pattern (Obsidian-vault-aware, env-overrideable). Easy follow-up.
